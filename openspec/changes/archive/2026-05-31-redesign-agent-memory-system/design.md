## Context

`agent_service` 当前的记忆系统位于 `internal/session/memory`，通过全局 `map[string]*SimpleMemory` 保存会话 ID 对应的最近消息窗口，默认最多保留 6 条消息。普通聊天和流式聊天在模型回复完成后把用户消息与助手回复写入该窗口，并在下一轮对话中作为 `history` 注入 prompt。

这个实现适合本地演示，但不适合团队级 AI 运维助手：

- 服务重启后记忆全部丢失。
- 记忆只按请求里的 `Id` 维度隔离，无法表达租户、团队、集群和会话的层级关系。
- 没有会话摘要，长对话只能被窗口截断。
- 没有长期记忆，不会沉淀稳定的环境约定、排障偏好和已确认事故结论。
- 没有长期记忆开关、删除、审计和来源追踪。
- 没有区分“固定指令”“会话上下文”“长期记忆”“知识库文档”和“实时监控事实”。

新的记忆系统需要借鉴 Claude Code 与 Codex 的分层思路：稳定规则应由人工维护，长期记忆应可治理、可删除、可审计，短期对话上下文应服务当前任务而不是无限增长。对于本项目，记忆的共享维度明确为团队、租户和集群，不以个人用户记忆作为核心目标。

## Goals / Non-Goals

**Goals:**

- 提供统一的 `MemoryManager`，让聊天入口不再直接依赖进程内 `SimpleMemory`。
- 支持租户、团队、集群共享的长期记忆，并保持不同作用域之间隔离。
- 支持短期会话窗口和滚动会话摘要，减少长对话被硬截断造成的信息丢失。
- 支持长期记忆显式开关，可按全局、租户、团队或集群维度控制写入与召回。
- 支持长期记忆查看、删除、按作用域清空，并同步清理持久化记录和向量索引。
- 支持记忆来源、置信度、创建方式、人工确认状态、创建时间、最后使用时间和过期时间等治理字段。
- 允许 `agent_service` 访问自己的持久化数据库保存记忆数据；监控事实仍然只通过 `api_gateway` 查询。
- 在 prompt 中有序注入固定指令、会话摘要、最近消息、长期记忆、知识库文档和实时监控事实。

**Non-Goals:**

- 不把监控事实源改为由 `agent_service` 直接访问业务 MySQL；监控事实边界保持不变。
- 不把普通内部文档知识库和长期记忆混成同一个概念；知识库仍用于文档 RAG，长期记忆用于助手工作上下文。
- 不在第一阶段实现复杂的多智能体协作记忆。
- 不在第一阶段实现跨租户或跨团队的全局共享记忆。
- 不把所有对话内容无条件写入长期记忆。

## Decisions

### 1. 采用租户、团队、集群、会话四级作用域

记忆上下文使用以下层级：

```text
tenant_id
  └── team_id
        ├── team-level memories
        └── cluster_id
              ├── cluster-level memories
              └── session_id
                    ├── recent turns
                    └── session summary
```

长期记忆默认只在 `tenant_id + team_id` 范围内共享；如果请求提供 `cluster_id`，则额外召回该集群的长期记忆。没有集群归属的运维偏好、输出格式和团队约定进入团队级记忆；涉及某个集群拓扑、历史问题、处置结论的内容进入集群级记忆。

替代方案是继续只用 `Id` 作为记忆键。该方案改动小，但无法支持团队共享、集群隔离和删除治理，因此不采用。

### 2. 记忆分为固定指令、会话窗口、会话摘要和长期记忆

新的上下文组装分为四类：

- 固定指令：人工维护的服务规则和团队约定，类似 `AGENTS.md` 或 `CLAUDE.md`，不由模型自动改写。
- 会话窗口：当前会话最近若干轮消息，服务短期上下文连续性。
- 会话摘要：长会话滚动总结，保留当前问题、已确认事实、已执行查询和待解决事项。
- 长期记忆：跨会话共享的稳定事实、偏好、集群知识和历史结论，可检索、可删除、可审计。

替代方案是只做向量化聊天历史召回。该方案容易把临时误解和未确认内容长期污染 prompt，也不利于删除与审计，因此不采用。

### 3. 引入 `MemoryManager` 作为唯一调用入口

聊天处理器和聊天编排层不直接操作具体存储，而是通过记忆管理器完成：

```go
type MemoryManager interface {
    LoadContext(ctx context.Context, scope MemoryScope, query string) (*MemoryContext, error)
    AppendTurn(ctx context.Context, scope MemoryScope, userMessage string, assistantMessage string) error
    SummarizeSession(ctx context.Context, scope MemoryScope) error
    ExtractDurableMemories(ctx context.Context, scope MemoryScope) error
    SetLongTermEnabled(ctx context.Context, scope MemoryScope, enabled bool) error
    DeleteMemories(ctx context.Context, selector MemorySelector) error
}
```

第一阶段可以保留内存短期窗口作为兼容实现，但聊天入口必须依赖接口而不是直接调用 `GetSimpleMemory`。这样后续切换 Redis、MySQL 或其他存储时不需要改动处理器。

### 4. 使用关系型数据库保存记忆元数据和正文，使用独立向量索引做语义召回

`agent_service` 新增自有持久化表保存长期记忆、会话摘要、开关配置和删除状态。语义召回使用独立的 Milvus collection，避免和运维文档知识库混用。

建议核心数据包括：

- `agent_memory_scopes`：保存租户、团队、集群级开关和策略。
- `agent_session_memories`：保存会话摘要、最近消息元信息和 TTL。
- `agent_long_term_memories`：保存长期记忆正文、类型、作用域、来源、置信度、确认状态、生命周期和向量 ID。
- `agent_memory_events`：保存创建、召回、更新、删除和清空事件，便于审计。

替代方案是只存 Milvus。该方案不适合做删除审计、开关策略、来源追踪和人工确认，因此不采用。

### 5. 长期记忆必须显式启用，并且写入与召回分别受控

长期记忆开关分为：

- 全局配置：控制服务是否启用长期记忆能力。
- 作用域配置：控制某个租户、团队或集群是否允许长期记忆写入。
- 召回配置：控制某个作用域是否允许长期记忆参与 prompt。

读取时按照“更具体作用域覆盖更宽作用域”的规则计算最终策略。这样可以支持某个团队启用长期记忆，但某个敏感集群禁用写入或召回。

### 6. 长期记忆采用候选抽取、规则过滤和确认状态

长期记忆不从每轮对话无条件写入。流程为：

```text
对话完成
  │
  ▼
抽取候选记忆
  │
  ▼
规则过滤与敏感内容检查
  │
  ├── 低风险、稳定、高置信度 → 自动写入为 active
  │
  └── 高风险、低置信度、可能敏感 → 写入为 pending 或直接丢弃
  │
  ▼
关系型数据库记录 + 向量索引
```

可自动写入的内容应限于稳定、非敏感、可复用的事实，例如团队偏好的报告格式、集群命名约定、已确认的历史故障结论。涉及凭据、个人信息、未确认猜测、一次性临时状态的内容不得自动写入长期记忆。

### 7. Prompt 组装采用固定顺序和预算控制

聊天上下文的推荐顺序：

```text
系统固定指令
  ↓
当前日期与边界说明
  ↓
会话摘要
  ↓
最近消息窗口
  ↓
长期记忆召回结果
  ↓
内部文档 RAG 结果
  ↓
用户当前问题
```

长期记忆召回必须有数量上限、token 预算和去重逻辑。召回结果应带上类型、作用域和置信度，便于 prompt 明确“这是团队记忆/集群记忆”，降低模型把记忆当作实时事实的风险。

### 8. 删除采用可审计删除流程，并清理向量索引

删除接口支持按记忆 ID、作用域、类型和时间范围选择目标。删除流程先把关系型记录标记为删除中，再删除向量索引，最后把记录标记为已删除并写审计事件。如果向量删除失败，记录保持删除中并由重试任务补偿；删除中的记忆不得参与召回。

替代方案是直接硬删所有数据。该方案审计能力弱，也不利于处理向量索引删除失败，因此不采用。对于确需合规硬删的场景，可以在后续增加物理清理任务。

## Risks / Trade-offs

- 记忆污染风险 → 通过候选抽取、规则过滤、确认状态、置信度和删除能力降低影响。
- 过期事实误导排障 → 为长期记忆设置 `expires_at`、`last_used_at` 和类型化生命周期，并在 prompt 中提示长期记忆不是实时监控事实。
- 多作用域召回混乱 → 在每条记忆中显式保存并注入作用域，召回时优先集群级，再补充团队级。
- 删除与向量索引不一致 → 使用删除中状态、重试补偿和召回过滤保证已请求删除的记忆不会再次被使用。
- 数据库依赖增加 → 将记忆存储封装在 `MemoryStore`，第一阶段提供清晰迁移和降级路径。
- Prompt 变长导致成本和延迟上升 → 对会话摘要、最近消息、长期记忆和文档 RAG 分别设置预算。
- 自动记忆涉及敏感内容 → 默认不保存凭据、个人信息和一次性临时状态，提供全局与作用域级禁用能力。

## Migration Plan

1. 新增记忆配置，默认保持长期记忆关闭，短期会话记忆行为与当前接口兼容。
2. 引入 `MemoryManager` 和内存兼容实现，把聊天入口从 `GetSimpleMemory` 迁移到统一接口。
3. 新增持久化表和独立向量 collection，但不立即开启长期写入。
4. 接入会话摘要和长期记忆召回，在配置关闭时不改变现有对话结果。
5. 新增管理接口和测试，支持查看、启停、删除和清空记忆。
6. 小范围开启长期记忆写入，观察候选质量、召回质量和删除一致性。
7. 稳定后逐步扩大启用范围。

回滚策略：关闭长期记忆全局配置后，聊天流程只使用固定指令、会话摘要和最近消息；如果持久化存储不可用，服务应退回短期内存记忆或直接无记忆运行，并返回可观测错误日志。

## Open Questions

- 第一版长期记忆是否需要人工确认 UI，还是先只提供管理 API 和 `pending` 状态。
- 会话摘要使用快速模型还是推理模型，需要根据成本和质量做一次小规模评估。
- 记忆数据库复用项目已有 MySQL，还是为 `agent_service` 独立配置数据库连接；设计倾向复用部署中的 MySQL，但表归属必须独立。
